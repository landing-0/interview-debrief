# interview-debrief

> 依赖：lark-cli（已安装）、lark-vc skill、lark-doc skill、lark-im skill

---

## 触发扩展：多轮面试模式

如果用户提到「这是同一家公司的第 N 轮」，或同时给出多场面试记录，**在完成单场分析后，额外执行第七步（多轮对比）和第八步（业务研究）**。即使只分析一场，只要用户提到公司名称且有下一轮，也应主动执行第八步。

---

## 第一步：权限检查

在拉取逐字稿前，先确认具备以下 scope。运行：

```bash
lark-cli auth status
```

必须包含：
- `vc:note:read`
- `minutes:minutes:readonly`
- `minutes:minutes.artifacts:read`
- `minutes:minutes.transcript:export`

如果缺少 minutes 相关 scope，先补授权再继续：

```bash
lark-cli auth login --scope "minutes:minutes:readonly minutes:minutes.artifacts:read minutes:minutes.transcript:export" --no-wait
# 把授权 URL 发给用户，等用户完成后继续
lark-cli auth login --device-code <device_code>
```

---

## 第二步：提取 minute_token

从用户提供的 URL 中提取 token（URL 最后一段路径）：

```
https://xxx.feishu.cn/minutes/obcXXXXXXXXX  →  obcXXXXXXXXX
```

---

## 第三步：拉取逐字稿

```bash
# 获取妙记的纪要产物（拿到逐字稿 doc_token）
lark-cli vc +notes --minute-tokens <minute_token>
```

返回结果中找到 `verbatim_doc`（逐字稿文档）的 token，然后：

```bash
# 读取逐字稿全文
lark-cli docs +fetch --doc <verbatim_doc_token>
```

逐字稿格式：`[说话人][时间戳] 内容`

**识别说话人角色：**
- 说话人数量 ≤ 2：多发言方 = 面试官，少发言方 = 候选人（本人）
- 说话人数量 > 2：结合称谓词（"你们公司"、"我们团队"）判断面试官 vs 候选人

---

## 第四步：AI 分析

读取完逐字稿后，按以下框架逐步分析，**所有评分需要有具体依据，不能给模糊分**：

### 4.1 基本信息提取
- 从逐字稿首尾推算时长
- 统计说话人数量 → 面试官数量
- 识别公司名、岗位（从问题内容推断，无法确定时留空让用户填）
- **说话时长比例**：利用逐字稿时间戳计算候选人 vs 面试官各自发言时长，得出候选人占比
  - 40–65% = 健康区间（双向交流充分）
  - >70% = 候选人可能过于主导/啰嗦
  - <35% = 候选人过于被动，或面试官兴趣不高
  - 如逐字稿无时间戳可忽略此项

### 4.2 问题全景（逐题）
遍历每一个面试官的提问，逐一评估：
- **问题类型**：技术深度 / 行为事件 / 情景假设 / 价值观 / 压力测试
- **回答质量 1-5**：
  - 5 = 有结构、内容准确、有深度、面试官没追问或正向追问
  - 4 = 基本准确，小缺失
  - 3 = 方向对但浅，或有结构但内容空
  - 2 = 明显知识缺失或逻辑混乱
  - 1 = 答非所问或直接说不会
- **是否被追问**：追问 = 面试官继续深挖同一话题
- **失分原因标签**（如有）：知识缺失 / 表达问题 / 无结构 / 深挖崩盘

### 4.3 表现量化
- **答题结构 /10**：是否使用框架（STAR/总分总/问题-方案-结果），散点式扣分
- **内容深度 /10**：能答到底层原理/数据支撑/边界条件
- **表达清晰 /10**：语言简洁、无过多口头禅、逻辑连贯
- **抗压表现 /10**：面对追问/质疑时是否稳定，有无慌乱或过度认错

本场核心短板：得分最低的 1-2 个维度，结合具体例子说明
本场亮点：表现最好的 1-2 个点，引用原文

### 4.4 错过的展示机会（Missed Opportunities）
面试官的提问方式往往隐含着 JD 诉求的关键词，候选人有时内容是对的但没有用对方的语言"呼应"，导致面试官感知不到匹配。

逐一检查：面试官每个问题里有没有隐含的岗位要求词（如"工程化思维"、"0到1能力"、"业务判断力"），候选人是否显式回应了这些词，还是只给了相关内容但没接住信号。

记录格式：
- 信号词："工程化思维"（面试官在 XX 问题中提及）
- 候选人是否接住：接住 / 部分接住 / 完全错过
- 如果错过，下次怎么补

### 4.5 候选人反问质量
候选人在面试中主动提问的质量，是成熟度和真实兴趣度的重要信号。

逐一列出候选人提出的每个问题，评分（1–5）：
- 5 = 深度问题：涉及业务目标、成功标准、团队挑战、技术方向等
- 3 = 中性问题：了解团队规模、工作流程等一般信息
- 1 = 低价值问题：薪资福利、上下班时间、假期等（留到 offer 阶段更合适）

给出整体印象：这批反问是否显示出候选人对岗位有深度思考，还是走流程式的礼貌发问？

### 4.6 面试官信号解读
- **被深挖的点**：面试官在同一话题连续追问 ≥ 2 次
- **一带而过的点**：面试官提问后候选人回答，面试官直接换题
- **面试官关注方向**：从高频词、深挖方向推断岗位核心诉求
- **潜在疑虑**：追问中带有质疑语气的点（"那你为什么……"、"这个有没有……"）
- **红旗信号**：以下任何一条出现即标记：
  - 频繁提到团队人员流动
  - 岗位职责描述模糊或边界不清
  - "需要能抗压/快速交付/独立扛事"等词频繁出现
  - 绩效/晋升话题被刻意回避

### 4.7 下一轮预测与准备
基于本轮面试官透露的信息（下一轮面试官身份、岗位核心诉求、本轮暴露的疑虑），预测下一轮大概率会遇到的问题类型，给出针对性准备建议：
- **下一轮面试官**：从本轮对话推断（如"业务VP，产研出身，技术深度要求高"）
- **大概率会问**：3–5个方向
- **本轮遗留疑虑需主动弥补**：本轮没答好但对方可能转给下一轮的点
- **建议准备的新素材/框架**

### 4.8 机会评估
基于面试官透露的信息（业务描述、团队介绍、发展空间等）打分：
- **业务成长性**：业务是否有增量，还是维稳/收缩
- **团队成熟度**：是否有清晰分工、成熟流程
- **技术氛围**：技术问题的深度和面试官本身的技术判断力
- **薪资回报**：如有提及，参考市场水位
- **与我匹配度**：岗位诉求 vs 候选人当前能力和职业阶段

### 4.9 通过率预测
先判断本轮面试官类型，再选对应权重模型：

**HR面（HRBP / HR负责人）**：技术深度她判断不了，核心看人
- 动机真实性 30%、沟通清晰度 25%、基本匹配度 25%、稳定性信号 20%

**技术面（直属技术Leader / 架构师）**：
- 技术能力 40%、项目经验 25%、结构化思维 15%、表达能力 10%、抗压稳定性 10%

**业务面（业务VP / 产品负责人）**：
- 业务理解 35%、项目匹配度 25%、思维框架 20%、沟通能力 10%、抗压稳定性 10%

**综合面（老板 / 合伙人）**：
- 价值观匹配 30%、成长潜力 25%、业务判断力 25%、表达影响力 20%

分维度逐一给分，加权得出综合通过率，标注主要风险点。

### 4.10 行动清单
- **P0**：本场暴露的、下轮大概率还会问的知识点/能力缺失，必须补
- **P1**：有提升空间、能显著加分的点
- **P2**：锦上添花，有时间再处理

### 4.11 能力资产沉淀
- **新增高频问题**：本场出现但以前没准备过的问题类型
- **可复用答案**：本场答得好的 1-2 个问题，提炼成精简版模板供下次用
- **能力标签更新**：哪些标签被本场验证（✅），哪些被本场暴露为弱点（❌）

### 4.12 一句话复盘
用一句话点出本场的本质问题（不是描述，而是诊断）：
例："本场本质问题是：技术细节能答出，但缺乏系统化思维，面试官想要架构感，你给的是实现细节。"

## 第七步：多轮对比分析（同一公司多场面试时执行）

拿到多场逐字稿后，跨轮对比：

### 7.1 公司考察模式识别
- **面试官层级变化**：从 HR → HR负责人 → 业务 VP → 老板，每个跳变意味着什么？
  - 连出两轮 HR：HR 话语权重，或候选人某个维度需要高层再确认
  - 技术面之前多轮 HR：传统企业，HR 做重筛，或该岗位曾频繁踩坑
  - 直接跳老板面：岗位优先级高，或团队规模小决策链短
- **各轮提问重叠度**：同一问题被反复问 = 公司对这个维度有疑虑，没被打消

### 7.2 公司真实诉求推断
把所有轮次面试官的关键词、深挖点、追问方向汇总，提炼出：
- 公司最担心的 1-2 件事（比如：稳定性、能不能自驱、能不能推动传统团队）
- 公司最想要的差异化能力（在所有轮次中被反复暗示的点）

### 7.3 候选人跨轮表现趋势
- 哪些点在多轮中持续加分？哪些问题被反复触发但没解决？
- 从第一轮到当前轮，候选人的状态是在改善还是在消耗？

## 第八步：业务研究与下一轮准备（知道公司名时执行）

### 8.1 搜索公司业务背景
```bash
# 用 WebSearch 搜索：
# 1. 公司名 + 主营业务/产品线
# 2. 公司名 + AI/数字化转型
# 3. 公司名 + 岗位名称（招聘JD）
```

结合逐字稿中面试官透露的业务信息（比如公司背景介绍、业务描述），综合还原：
- 公司的客户群体（to B / to C / to 小b）
- 核心商业模式（卖什么、怎么赚钱）
- 当前 AI 转型处于哪个阶段（0-1 探索 / 1-N 规模化 / 内部提效为主）

### 8.2 AI 场景推断
基于公司业务，推导 2-3 个具体的 AI 落地场景：
- 说清楚**场景是什么**（用户/员工的什么问题）
- **AI 怎么解决**（用什么方式，不需要很技术，说清逻辑就行）
- **对应的核心指标**是什么

### 8.3 候选人经验的「平移」
找候选人过往经历中，和该公司场景**结构最相似**的项目：
- 两者的共同点是什么（比如：都是帮用户在大量 SKU 里匹配最合适的选项）
- 候选人可以如何把自己的经验「翻译」成对方听得懂的语言
- 明确说出：「你在 XX 做的 YY，就是这家公司需要在 ZZ 场景里做的事」

### 8.4 下一轮面试官类型预判 + 针对性准备
- 根据面试进程判断下一轮最可能是谁（技术 Leader / 业务 VP / 老板）
- 他们会重点考察什么（结合 7.2 公司诉求）
- 给出 2-3 个建议提前准备的具体问题和回答方向

## 注意事项

1. **逐字稿识别失败时**：提示用户手动提供面试官姓名或让用户确认哪个说话人是面试官
2. **公司/岗位无法推断时**：在文档中留空，并在 IM 消息中提示用户手动补充
3. **逐字稿过长时**：分段处理，确保所有问题都被覆盖，不遗漏
4. **无逐字稿权限时**：提示用户补充对应 scope 后重试，或让用户手动粘贴文字记录
